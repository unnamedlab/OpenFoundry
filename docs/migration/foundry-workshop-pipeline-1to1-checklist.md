# Foundry Workshop and Pipeline Builder 1:1 parity checklist

Date: 2026-05-10  
Scope: public-docs-based parity plan for OpenFoundry's Pipeline Builder,
geospatial surface, Data Connection webhooks, Ontology actions, Functions, and
Workshop-style application builder.

This document is intentionally implementation-oriented. It does not attempt to
clone Palantir branding, private source code, proprietary assets, or any
non-public behavior. The target is **functional parity based on public Palantir
documentation**: same product concepts, same builder workflows, compatible
resource models where useful, and a user experience close enough that the
Trail Running demo can be built natively inside OpenFoundry.

## Parity scope boundary

All checklist work is governed by the
[Foundry public-docs parity policy](../reference/foundry-public-docs-parity-policy.md).
OpenFoundry may implement behavior described in public Palantir documentation,
but contributors must not copy private source, decompile bundles, import
tenant-specific exports, use Palantir branding, or reuse proprietary assets.
The product target is functional parity in an OpenFoundry-native implementation,
not a pixel-perfect clone.

## Status vocabulary

| Status | Meaning |
| --- | --- |
| `todo` | Not implemented or not yet verified in OpenFoundry. |
| `partial` | Some surface exists, but behavior is incomplete or not wired end-to-end. |
| `blocked` | Requires a platform dependency, public documentation, or product decision. |
| `done` | Implemented, tested, documented, and verified through UI or API smoke tests. |

## Priority vocabulary

| Priority | Meaning |
| --- | --- |
| `P0` | Required to build the Trail Running demo end-to-end. |
| `P1` | Required for credible Foundry-style parity beyond the demo. |
| `P2` | Advanced or scale-oriented parity. |

## Trail Running demo target architecture

```text
Strava JSON files
  -> Pipeline Builder / Python transform
  -> RunActivity dataset
  -> RunActivity object type

GPX trail files
  -> GPX parser transform
  -> Trail dataset with GeoJSON LineString and trailhead GeoPoint
  -> Trail object type

Coffee shop table
  -> Dataset import
  -> CoffeeShop object type with GeoPoint

Trail + CoffeeShop
  -> Haversine / nearest-neighbor transform
  -> TrailCoffeeRecommendation link/output table

Trail + RunActivity
  -> Effort estimator function or transform
  -> TrailEstimate object type

Open-Meteo REST API
  -> Data Connection REST source
  -> Webhook
  -> WeatherSnapshot object type through Ontology action

Workshop app
  -> Object Table, Filter List, Property List, Chart XY, Metric Card, Map, Button Group
  -> Published runtime app
```
